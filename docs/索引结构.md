db 目录下基本上就是“一套可跑通的 KV 引擎实现”（`WAL` + `memtable` + `flush/compaction` + `L0/L1` + `manifest/version` + 存储分配器），对外接口在 `include/db.h` 声明、在 `db/db.cpp` 与 `db/db_client.cpp` 实现；它不是“只定义接口、必须依赖外部 KV 引擎才能工作”的那种结构。

但它的“索引结构”是可插拔的：引擎内部统一依赖抽象索引接口 `Index`（见 `include/db_common.h`），具体实现来自本仓库的 `lib/`（可看作“引擎内置/集成的第三方索引库封装”，而不是外部 KV 引擎）：

- `MasstreeIndex` 定义在 `lib/index_masstree.h`，`memtable` 与 `L0/L1` 都会用到（例如 L1 在 `db/compaction/version.cpp`，L0 tree 在 `db/compaction/version.cpp`）。
- `HOTIndex` 定义在 `lib/index_hot.h`，也可用于 `memtable`/`L0`/`L1`（同样在 `db/compaction/version.cpp` 有条件编译分支）。**注意**：该实现里 `Delete` 目前是空实现（见 `lib/index_hot.h`），所以选它可能意味着删除语义不完整。

因此更准确的结论是：

1. 不需要外部 KV 引擎；`db/` 本身就是 KV 引擎主体实现。
2. 部分能力依赖（集成在仓库里的）外部索引库/实现，通过 `Index` 抽象接入；用哪个由编译宏控制（例如 `include/config.h` 的 `MASSTREE_MEMTABLE`/`MASSTREE_L1` 等）。
3. 同时它确实存在“工程上不完全通用”的限制（例如很多路径假设 8-byte key/value、某些宏组合才支持 value 分离等），所以可视为“可配置、但在若干配置上仍不完整”的 KV 引擎。

### 1) 接口契约：Index 是引擎内部唯一依赖的“索引能力面”
抽象接口定义在 `include/db_common.h`：

**必须实现的核心虚函数**：
- `ThreadInit(int)`
- `Get(KeyType)`
- `Put(KeyType, ValueHelper&)`
- `PutValidate(KeyType, ValueHelper&)`
- `Delete(KeyType)`

**可选扫描能力**（基类默认 `ERROR_EXIT`，所以如果上层调用了而你没实现，会直接崩）：
- `Scan(...)` / `Scan2(...)` / `ScanByRange(...)`：见 `include/db_common.h`

这意味着：`Index` 的“可插拔”不是通过模板，而是标准的运行时多态（虚函数）；上层完全不知道你是 Masstree、HOT、CCEH 还是别的结构。

---

### 2) 插拔点：哪些地方把具体索引 new 出来
#### 2.1 memtable 索引（用户写路径使用）
MYDB 里维护 `mem_index_[]`（类型是 `Index*`），初始化/切换时会创建具体实现：

- 初始创建在 `db/db.cpp`
  ```cpp
  #ifdef MASSTREE_MEMTABLE → mem_index_[...] = new MasstreeIndex();
  #ifdef HOT_MEMTABLE → mem_index_[...] = new HOTIndex(...);
  ```
- flush 切换 active memtable 时重新 new 一个 index（下一张 memtable）在 `db/db.cpp`

这里的“插拔”就是：改宏 → 编译时决定 new 哪个 `Index` 实现。

#### 2.2 L0 / L1 的“表索引”（查找 PST 用）
Version 内部也用 `Index*` 维护：
- `level1_tree_`：L1 的索引（用来从 key 快速定位到 `level1_tables_` 中的表 id）
- `level0_trees_[]`：每个 L0 tree 的索引（用来定位该 tree 下某些 PST）

**创建点**：
- L1 index 的创建在 `db/compaction/version.cpp`
  ```cpp
  #ifdef MASSTREE_L1 → level1_tree_ = new MasstreeIndex();
  #ifdef HOT_L1 → level1_tree_ = new HOTIndex(MAX_PST_L1);
  ```
- 每次新增 L0 tree，会 new 一个 index 实例在 `db/compaction/version.cpp`

---

### 3) 上层如何“只通过 Index 调用”，从而实现解耦
#### 3.1 Put/Insert：上层只调用 Index::Put / PutValidate
例如 `Version::InsertTableToL0` 中，更新 L0 tree 索引只依赖 `Index::Put`：

```cpp
// db/compaction/version.cpp
ValueHelper lh((uint64_t)idx);
level0_trees_[tree_idx]->Put(table.max_key_, lh);
```

同理 `InsertTableToL1` 也是 `level1_tree_->Put(...)`，并依赖 `ValueHelper` 的 `old_val` 来处理“替换同 key 的旧表 id”（见 `db/compaction/version.cpp`）。

这说明：索引对上层输出的“附加语义”主要靠 `ValueHelper` 回传，而不是改接口。

#### 3.2 Get/Scan：上层用 Index::Get / ScanByRange 等虚函数
- `MYDBClient::GetFromMemtable` 会对 `mem_index_[memtable_id]->Get(...)` 调用（实现点在 `db/db_client.cpp` 附近）
- flush 会调用 memtable 的扫描接口，比如 `ScanByRange`：见 `db/compaction/flush.cpp`
- version 在构造 L1 iterator/查表时会扫描索引来拿 table id（相关调用点在 `db/compaction/version.cpp`）

所以：你的新 `Index` 实现要么实现这些 `Scan/ScanByRange`，要么确保上层不会走到这些路径（否则基类默认 `ERROR_EXIT` 会直接退出）。

---

### 4) 现有实现如何体现“插拔”，以及注意事项
仓库里已经有 3 个 `Index` 实现（都在 `lib/`）：

- `MasstreeIndex`：`lib/index_masstree.h`（实现了 `Get/Put/PutValidate/Delete/Scan2/ScanByRange` 等）
- `HOTIndex`：`lib/index_hot.h`（实现了 `Get/Put/Scan/Scan2/ScanByRange`，但 `Delete` 目前空实现，见 `lib/index_hot.h`）
- `CCEHIndex`：`lib/index_cceh.h`（`Get/Put` 有实现，`Delete` TODO，扫描未实现）

配置开关主要来自 `include/config.h`（memtable/L1 选 Masstree 等），另外 `HOT_L1/HOT_MEMTABLE` 这类宏看起来可能由 CMake 控制（因为 `config.h` 里只显式定义了 Masstree）。

---

### 5) 如果你要接入“其他索引结构”，需要做什么（按这份代码的实际用法）
**最小可行路径**：

1. 新增一个 `lib/index_xxx.h`，实现 `class XXXIndex : public Index`（参考 `lib/index_masstree.h`）
2. 确保实现了上层会用到的方法：
   - memtable：至少 `Get/Put(或 PutValidate)/Scan2/ScanByRange`（flush 用到）
   - L0/L1：至少 `Put/Get/Delete`，以及 version 查找可能需要的扫描（取决于 `ScanIndexForTables` 的实现细节）
3. 在 `db/db.cpp` 与 `db/compaction/version.cpp` 的宏分支里，把 `new MasstreeIndex()` 替换成 `new XXXIndex()`（或添加新宏分支）


继续把“索引如何通过 Index 插拔”的细节补全后，可以看到一个很关键的结论：

- memtable 的插拔基本是“纯 Index 接口多态”：上层只调用 Get/Put/PutValidate/ScanByRange/Scan2 等。
- L0/L1 的插拔在 `MASSTREE_L1` 分支下同样是“纯 Index 接口多态”，但在 `HOT_L1` 分支下，代码里直接 downcast 到 `HOTIndex` 并调用 `Seek()`——这会导致“插拔并非完全通过 Index 接口”，而是存在 HOT 的专用 fast-path。

下面结合具体代码讲清楚。

---

### 1) Index 接口层面：上层依赖哪些能力
接口定义在 `include/db_common.h`。

**必需能力**（纯虚函数，上层一定会用到）：
- `Get` / `Put` / `PutValidate` / `Delete` / `ThreadInit`

**可选扫描能力**（默认实现是 `ERROR_EXIT`，一旦走到会直接退出）：
- `Scan`：见 `include/db_common.h`
- `Scan2`：见 `include/db_common.h`
- `ScanByRange`：见 `include/db_common.h`

这决定了：要能被“安全插拔”，你的索引实现必须实现“实际会被调用到”的扫描方法。

---

### 2) memtable 场景：严格通过 Index* 调用（可插拔性最好）
#### 2.1 创建点（决定用哪种索引）
MYDB 在初始化与 flush 切换 memtable 时创建具体索引：

- 初始化 active memtable：在 `db/db.cpp`
  ```cpp
  #ifdef MASSTREE_MEMTABLE → new MasstreeIndex()
  #ifdef HOT_MEMTABLE → new HOTIndex(...)
  ```
- flush 后切换到下一张 memtable：在 `db/db.cpp`

这里的插拔就是：宏决定 new 哪个实现，但对外都当成 `Index*`。

#### 2.2 被调用的接口集合（决定实现必须提供什么）
- flush 会对 memtable 扫描分区范围：`ScanByRange(...)`（见 `db/compaction/flush.cpp`）
- scan 也会对 memtable 做 `Scan2`：见 `db/db_client.cpp`（你前面已读到这一段）

因此：如果你想替换 memtable 索引，至少要实现 `Get/Put(或 PutValidate)/Scan2/ScanByRange`，否则 flush/scan 会崩。

---

### 3) L0/L1 “表索引”场景：表面上用 Index，但存在 HOT 专用路径
Version 用索引来做“从 key → PST 表 id”的定位（索引里存的 value 实际是 table 的 vector 下标）：

> 插入时把 `max_key_` 写进索引、value 写成 `idx`：见 `db/compaction/version.cpp` 和 `db/compaction/version.cpp`

#### 3.1 查表辅助函数：FindTableByIndex / ScanIndexForTables
关键点在 `db/compaction/version.cpp`：

**`FindTableByIndex(key, Index* level_index)`**：
- `MASSTREE_L1` 分支：调用 `level_index->Scan(key, 1, temp)`，取 `temp[0]`（见 `db/compaction/version.cpp`）
- `HOT_L1` 分支：`reinterpret_cast<HOTIndex*>(level_index)->Seek(key)`，取 iterator 的 `Value()`（见 `db/compaction/version.cpp`）

**`FindTableByIndex2`**（类似，但要取“第二个”候选表）：
- Masstree 用 `Scan(key,2)`，取 `temp[1]`
- HOT 用 `Seek(key)` 后 `Next()` 再取 `Value()`（见 `db/compaction/version.cpp`）

**`ScanIndexForTables(key, level_index, table_num, output_table_ids)`**：
- 目前只在 `MASSTREE_L1` 分支下实现，直接调用 `level_index->Scan(key, table_num, output_table_ids)`（见 `db/compaction/version.cpp`）
- 并且 `GetLevel1Iter` 会依赖 `ScanIndexForTables` 拿到候选表 id 列表（见 `db/compaction/version.cpp`）

#### 3.2 结论：L0/L1 的“插拔”存在两种模式
- **Masstree 模式（`MASSTREE_L1`）**：完全通过 `Index::Scan` 实现定位候选表 → 可替换为任何实现了 `Scan` 的 `Index`。
- **HOT 模式（`HOT_L1`）**：Version 里写死了 `HOTIndex::Seek`，这不是 `Index` 接口的一部分 → 你想换成“别的索引结构”，必须：
  1. 改 Version 代码把这段 HOT 专用逻辑抽回 `Index` 接口（比如用 `Scan`），或者
  2. 让你的新索引类也叫 `HOTIndex` 且提供 `Seek`（不现实/不优雅），或者
  3. 新增一个接口（例如在 `Index` 里加 `SeekLowerBound` 之类）并改所有实现。

所以：“通过 Index 插拔”在 L0/L1 上并不完全成立；它在 `MASSTREE_L1` 才是纯接口插拔，在 `HOT_L1` 会变成“接口 + 具体类型特化”。

---

### 4) 对“其他索引结构”如何接入的建议（结合现状）
如果你要接入新的索引（比如 BwTree/ART/自研 B+Tree）并希望做到真正可插拔，最省事的路径是：

1. 先走 `MASSTREE_L1` 这条抽象路径：实现好 `Index::Scan`（至少支持 `Scan(key, n, vec)` 返回从 key 开始的 n 个 value），这样 `FindTableByIndex/FindTableByIndex2/ScanIndexForTables` 不用动。
2. 如果一定要像 HOT 那样用 lower_bound iterator 做高效定位，那么目前代码结构需要调整：把“迭代器式 lower_bound”抽象进 `Index`（否则 Version 里会不断出现对具体实现的 `reinterpret_cast`）。
