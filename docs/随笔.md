# 模块划分与核心流程（db/ 目录）

## 一、模块划分（db/ 目录）
| 模块功能 | 对应文件/路径 | 核心说明 |
|----------|---------------|----------|
| 入口/对外实现层 | `db/db.cpp`、`db/db_client.cpp`、`include/db.h` | - `db/db.cpp`：MYDB 实例管理 + 后台任务（Flush/Compaction）调度<br>- `db/db_client.cpp`：面向用户线程的读写 API 实现<br>- `include/db.h`：对外 API 声明 |
| 日志（WAL / Value 存放） | `db/log_writer.cpp`、`db/log_reader.cpp`、`db/allocator/segment.h` | - 写入：`LogWriter::WriteLogPut/Delete`<br>- 回读：`LogReader::ReadLogForValue`<br>- 存储载体：`LogSegment`（allocator 管理） |
| 存储与空间管理 | `db/allocator/segment_allocator.h`、`db/allocator/segment.h` | - 核心：统一 segment 池 + 位图分配<br>- 两类 Segment：`LogSegment`（日志）、`SortedSegment`（PST/DataBlock 页面） |
| PST（持久化有序表）与 DataBlock | `db/table.h`、`db/pst_builder.cpp`、`db/pst_deleter.cpp`、`db/pst_reader.cpp`、`db/datablock_writer.cpp`、`db/datablock_reader.cpp` | - 元信息：`db/table.h`<br>- 构建/回收/读取：对应 builder/deleter/reader<br>- 底层块读写：datablock_writer/reader |
| Flush/Compaction 与元数据 | `db/compaction/flush.cpp`、`db/compaction/compaction.cpp`、`db/compaction/version.h`、`db/compaction/manifest.h` | - Flush：`FlushJob`（memtable → L0 PST）<br>- Compaction：`CompactionJob`（L0→L1 合并）<br>- 版本视图：`Version`（内存元数据）<br>- 持久化元数据：`Manifest`（manifest + flush log） |

## 二、模块间如何交互（典型链路）
### 1. Put/Delete 写路径（`db/db_client.cpp` + `db/log_writer.cpp`）
1. `MYDBClient::Put/Delete` 触发写操作；
2. `LogWriter::WriteLogPut/Delete` 追加日志并返回 `log_ptr`；
3. 更新 memtable index（按宏分支处理）：
   - `INDEX_LOG_MEMTABLE`：写入 `ValuePtr` 指向日志；
   - `BUFFER_WAL_MEMTABLE`：直接存值/指针；
4. `MYDB::MayTriggerFlushOrCompaction` 依据 memtable 大小/L0 数量，触发后台 `BGFlush`/`BGCompaction`（`db/db.cpp`）。

### 2. Get 读路径（`db/db_client.cpp`）
1. 优先 `GetFromMemtable`：从最新 memtable 往旧的查询，命中日志指针则通过 `LogReader::ReadLogForValue` 回读值；
2. 未命中则调用 `Version::Get`（`db/compaction/version.h`）；
3. 通过 `PSTReader`/`DataBlockReader` 做点查（`db/pst_reader.cpp`、`db/datablock_reader.cpp`）；
4. 若启用 `KV_SEPARATE`，`Version` 层返回 value 指针，最终需 `LogReader` 回读真实 value。

### 3. Scan 范围读（`db/db_client.cpp`）
1. `MYDB::WaitForFlushAndCompaction` 暂停，等待后台任务结束；
2. memtable 调用 `Scan2` 收集多路结果，同时 `Version::GetLevel1Iter` 得到 L1 迭代器；
3. 用小顶堆做多路归并（重复 key 去重/覆盖规则由“先弹出的来源”隐式决定）。

## 三、后台流程（Flush/Compaction/Recovery）
### 1. Flush（memtable 固化为 L0 PST）
1. `MYDB::BGFlush` 切换 active memtable（新 memtable 承接写入）；
2. `FlushJob::subrunParallel` 按 partition 扫描 memtable index（`ScanByRange`）；
3. 必要时通过 `LogReader` 从日志取回 value；
4. `PSTBuilder`/`DataBlockWriter` 写入 DataBlock 页，生成 `PSTMeta`（含 `key_256/512/768` 等采样点，`db/table.h`、`db/pst_builder.cpp`）；
5. `Manifest::AddTable` 持久化元数据，`Version::InsertTableToL0` 更新内存视图；
6. 删除/标记回收本次 log group 的 `LogSegment`（flush log 保障崩溃一致性，`db/compaction/flush.cpp`、`db/compaction/manifest.h`）。

### 2. Compaction（L0→L1 合并）
1. `MYDB::BGCompaction` 触发合并任务；
2. `CompactionJob::PickCompaction` 选取若干 L0 tree + 与之重叠的 L1 表；
3. `RunSubCompactionParallel` 按 partition 并行归并，输出新 PST；
4. `CleanCompactionWhenUsingSubCompaction` 删除旧 PST，更新 `Version`/`Manifest`（`db/db.cpp`、`db/compaction/compaction.cpp`）。

### 3. Recovery（启动恢复）
1. allocator 恢复位图/日志段列表（`db/allocator/segment_allocator.h`）；
2. `Manifest` 恢复 `Version` 视图（`db/compaction/manifest.h`）；
3. MYDB 扫描残留日志，通过 `RecoverLogAndMemtable` 重建 memtable（`db/db.cpp`，主逻辑入口）。

## 四、对外提供的接口（及实现落点）
### 1. 核心对外 API（`include/db.h`）
`MYDB` / `MYDBClient` 提供的核心接口：
- 实例管理：`Open`/`Close`；
- 客户端管理：`CreateClient`；
- 数据操作：`Put`/`Delete`/`Get`/`Scan`。

### 2. API 实现落点
| 接口功能类型 | 实现文件 | 核心职责 |
|--------------|----------|----------|
| 实例生命周期/后台调度 | `db/db.cpp` | 实例创建/销毁、`BGFlush`/`BGCompaction` 调度、模式切换 |
| 线程态读写/扫描 | `db/db_client.cpp` | 读写路径执行、日志写入与 memtable 更新、scan 多路归并 |
